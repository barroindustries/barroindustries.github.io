
`allow read: if isAuth()` — this DENIES the exact query above, because the query runs BEFORE `auth.signInWithEmailAndPassword`, i.e. while `request.auth == null`. The `.where('username','==',...)` query throws a permission-denied and is NOT caught (no `.catch()` on that specific await) — for a genuine username-login attempt this throws inside the try block and is swallowed by the outer `catch(err)`, which calls `friendlyError(err.code)` (app.js:759-767) — `permission-denied` isn't in that map, so it falls through to the generic `'Sign-in failed.'` message. Net effect: every worker who was given a username (via "Create Worker Account", see below) cannot ever log in — the lookup silently 403s and shows a generic error, not the "No account found" or "Account not configured" messages that only fire on a *successful* read.

Precedent for a narrow public-read pattern already in this codebase — firestore.rules:795-807 (`order_tracking`, quoted verbatim):
```
    // Each doc id is a random 20-char Firestore auto-id acting as an unguessable
    // access token — the shareable client link is /track.html?t={id}. `get` is
    // PUBLIC (no auth) so a client can open the link without an account; `list` is
    // DENIED so nobody can enumerate all orders. Only client-SAFE fields live here
    // (status, dates, amounts the client already knows) — never internal cost/margin.
    // Internal non-partner staff create/update it; admins may delete.
    match /order_tracking/{token} {
      allow get:    if true;
      allow list:   if false;
      allow create: if isAuth() && getRole() != 'partner';
      allow update: if isAuth() && getRole() != 'partner';
      allow delete: if isAuth() && isAdmin();
    }
```
Note the shape that makes that pattern safe: `get` (single-doc, known-id) is public, `list`/query is explicitly denied. The username lookup is the OPPOSITE shape — it's a `.where()` query (a `list` operation), which Firestore rules cannot narrow to "only your own username" the way `get` can be narrowed to "only if you know the id" — a public `allow list: if true` on `/users` would let anyone enumerate every username, display name, role, department, employeeId, etc. in the entire company. This is why the two options in openDecisions are meaningfully different in shape, not just style.

## Constraints — must respect

- Script load order is fixed and load-bearing (CLAUDE.md, confirmed unchanged): index.html loads Firebase SDK+Chart.js+Lucide → firebase-config.js → config.js → drive.js → notifications.js → departments.js → app.js → modules.js, all deferred, all via window globals — no ES modules. Any new file (e.g. a QR-generation helper) must be added to BOTH index.html's script list and sw.js's PRECACHE array, and must be positioned so it's defined before whichever of app.js/departments.js/modules.js calls it.
- CACHE_VER in sw.js (currently 'bi-ops-v162', sw.js:11) MUST be bumped on every JS/CSS edit — this repo's pre-commit hook auto-bumps window.APP_VERSION in js/config.js and the vX.Y.Z strings in index.html, but CACHE_VER itself is a separate manual bump (per CLAUDE.md) — do not hand-edit APP_VERSION, but DO bump CACHE_VER by hand when touching js/app.js, js/departments.js, css/styles.css for this workstream.
- Every Firestore collection needs an explicit rules match — there is no cascade/prefix matching in these rules (confirmed pattern throughout firestore.rules: worker_profiles, order_tracking, users, leave_balances, gov_biddings are all separately enumerated). A missing rule denies silently unless the client `.catch()`s it (e.g. `.catch(()=>({docs:[]}))`, seen throughout departments.js) — any new collection this workstream introduces (e.g. a worker-ID public-verify collection, or a `_counters/workers` doc) needs its own explicit match block AND `firebase deploy --only firestore:rules` (git push does NOT deploy rules — CLAUDE.md, confirmed).
- escHtml() must wrap every field interpolated into innerHTML — confirmed as the universal convention in renderIDCard, renderTeam, renderFinanceHRProfiles, openHRProfileForm (every `${escHtml(...)}` call). Any new HTML (worker ID card template, batch-print sheet) must follow the same discipline, including for `worker_profiles` fields which are HR-typed free text (idNumber, name, address, phone, ssNum/phNum/pagibigNum/tinNum) and therefore just as injectable as user-typed content.
- Manila-time discipline: `issuedOn`/`startDate` on both `users` and `worker_profiles` docs are populated via `bizDate()` (app.js:452, departments.js:3713 `today()` alias) — never raw `new Date().toISOString()`. Any new 'issued on' stamp for a worker ID card must use the same helper.

### Resolved decisions

1. **Worker-login username unblock → OWNED BY WS19, not re-decided here.** WS19 (DECIDED) already ships the public `usernames/{usernameLower}` → `{email, uid}` map (`allow get: if true; allow list: if false; create/update/delete: isAuth() && isAdmin()`) and rewrites `app.js:696-703` to `db.collection('usernames').doc(input.toLowerCase()).get()`. Rationale: composing beats a second competing pre-auth pattern.
2. **QR-verify uses a NEW public `id_verify/{token}` mapping collection — never `worker_profiles`/`users` directly.** Mirrors the `order_tracking` precedent (`get:true`/`list:false`, unguessable token). SSS/PhilHealth/Pag-IBIG/TIN/rates/caBalance/address/phone/email NEVER leave the sensitive doc. One collection serves BOTH employee IDs and worker IDs via a `kind` discriminator. Rationale: keeps every payroll field one layer away from any public rule — a narrow rule on `worker_profiles` itself is too dangerous given it co-houses SSS/TIN/CA balance.
3. **`worker_profiles` and `hrManagedAccount` users stay TWO separate identities** — with an OPTIONAL one-way `worker_profiles.linkedUserId` (nullable) set only when HR explicitly issues a login for that worker. No change to `salary_raises.subjectType`, no change to the payroll-vs-worker_profiles pay split. Rationale: unifying reopens the "two sources of truth" class-issue flagged in finance memory and collides with WS22/23/24 which already key on the split; a soft link is enough for the ID system, which discriminates by `kind` exactly as `salary_raises` discriminates by `subjectType`.

```js
function buildIdVerifyDoc(kind, src, uidOrNull) {
  const B = window.BRAND || {};
  const dept = Array.isArray(src.departments) && src.departments.length
    ? src.departments.join(', ') : (src.department || '');
  return {
    kind,                                            // 'employee' | 'worker'
    name:           src.displayName || src.name || '',
    photoUrl:       src.photoUrl || '',
    idNumber:       src.employeeId || src.idNumber || '',

// Ensure the LOGGED-IN employee's own verify token exists; idempotent (reuses
// an existing token so reprinted QR codes stay stable). Returns the token.
async function ensureEmployeeVerifyToken(u) {
  if (u.verifyToken) {
    // refresh the public projection in case name/photo/dept changed
    db.collection('id_verify').doc(u.verifyToken)
      .set(buildIdVerifyDoc('employee', u, currentUser.uid), { merge: true }).catch(()=>{});
    return u.verifyToken;
  }
  const token = window.makeTrackCode(10);
  await db.collection('id_verify').doc(token)
    .set(buildIdVerifyDoc('employee', u, currentUser.uid));
  await db.collection('users').doc(currentUser.uid)
    .set({ verifyToken: token }, { merge: true });      // not a frozen field → self-write OK
  u.verifyToken = token;
  if (window.userProfile && userProfile.id === currentUser.uid) userProfile.verifyToken = token;
  return token;
}
```

- ⚠️ Security: any rule loosening on `/users/{uid}` to unblock the pre-auth username lookup must not accidentally widen the surface for the ALREADY-DENY-BY-DEFAULT `list` operation on the full roster — a naive `allow read: if true` (rather than a narrowly-scoped `get`/token collection) would let an unauthenticated caller enumerate every employee's name, role, department, employeeId, and phone number company-wide, which is a strictly worse posture than today's silent-deny bug.
- ⚠️ Interaction with workstream 22 (Cash-advance)/23 (Raises)/24 (Payslip) — all three explicitly reference the same `worker_profiles` collection and the same `salary_raises`/`subjectType:'worker_profile'` records this workstream's ID-card feature reads from; a schema change here (adding `photoUrl`, an auto BI-W-#### `idNumber`) needs coordinating with whichever workstream lands the 'ONE CashAdvance service' / raise-approval-routing changes so field additions don't get silently overwritten or duplicated by two build passes touching the same doc shape in parallel.
- ⚠️ Migration hazard: existing `worker_profiles.idNumber` values are free-text (HR-typed, inconsistent format) — introducing an auto-incrementing `BI-W-####` counter needs a decision on whether to backfill existing profiles (renumbering already-issued/possibly-already-printed physical IDs) or only apply the new counter to profiles created after this workstream ships, leaving old ones on their legacy free-text idNumber permanently.
- ⚠️ The `hrManagedAccount: true` flag (set only by openCreateWorkerModal, app.js:6722) is already relied upon as a security gate elsewhere — `adminResetPassword` (functions/index.js:203-211) explicitly refuses to reset a password unless the target doc has `hrManagedAccount:true` ('Only HR-managed worker accounts can be reset here'). Any 'worker login unblock' fix must not bypass or weaken this check, and any new username-lookup Cloud Function should probably enforce the same `hrManagedAccount` (or equivalent `username` presence) precondition to avoid becoming a general-purpose unauthenticated email-lookup oracle for EVERY user in the company, not just HR-managed worker accounts.
- ⚠️ renderTeam()'s own gate (`if(!isPresident()&&currentRole!=='manager')`) is stricter than the HR hub's gate (`['president','manager','secretary','finance'].includes(role)`, departments.js:2093) — if Create-Worker-Account is re-linked from inside the HR hub (reachable by 'secretary'/'finance'), either renderTeam()'s own access check needs updating to match, or the new nav entry needs to be conditionally hidden for secretary/finance so it doesn't produce a dead click into `renderAccessDenied('Team')`.
- ⚠️ CACHE_VER must be bumped for this workstream's JS/CSS edits (app.js, departments.js, styles.css, possibly a new standalone verify page) — forgetting it (an explicit, repeatedly-flagged failure mode in this repo's own memory/CLAUDE.md) will make the change appear broken/not-deployed to users behind the service worker's cache-first strategy.
- ⚠️ If Fable's chosen fix touches `firestore.rules`, it must be deployed with `firebase deploy --only firestore:rules` SEPARATELY from `git push origin master` — the latter does not deploy rules (explicit repo convention, confirmed in CLAUDE.md and the user's own working memory) — a rules-only fix that's just committed and pushed will silently NOT take effect until someone remembers the separate deploy command.
- ⚠️ If the worker ID-card verify page introduces a new public HTML entry point (like track.html), it needs its own PRECACHE/service-worker consideration and, per this repo's memory on Firestore rules collection coverage, an explicit new match block — reusing the order_tracking collection/rule for a structurally different purpose (people, not orders) would conflate two access-control domains that should stay separate.
- ⚠️ Print/export CSS is being built from zero (no `@media print` precedent in this codebase) — CR80 dimensions must be gotten exactly right for physical card printing (CR80 = 3.375in × 2.125in / 85.60mm × 53.98mm) since there's no existing print-scaled component in this app to copy proportions from, unlike the quote-builder's recently-fixed print pagination (aab024a) which is a different file (quote-builder-v2.html) with unrelated print CSS.
